## 2025-02-20 - Unrolling 3x3 Convolution Kernels (Sobel) in Java
**Learning:** In Java, flattening small kernel loops like 3x3 Sobel filters directly into unrolled arithmetic yields a dramatic speedup (often 2.5x to 4x). The overhead from a nested 3x3 loop, including bounds-checking and arithmetic for computing array indexes (e.g. `(r+kr)*pW+(c+kc)` vs pre-calculated row offsets `row0 + c`), is surprisingly large relative to the actual math being performed.
**Action:** When identifying mathematical kernels like Sobel filters in image or heightmap processing loops, unroll the 3x3 array accesses directly, use local row offset variables, and multiply by the inverse (e.g., `* 0.125f`) instead of dividing.

## 2025-02-20 - Optimizing Bilinear Interpolation
**Learning:** In Java image processing, pre-calculating interpolation coordinates/weights outside of the horizontal (inner) loop and caching the arrays saves huge amounts of repetitive float arithmetic and Math.floor calls per pixel.
**Action:** When resizing arrays/images or doing interpolation (e.g., bilinearResize), allocate small 1D arrays for the inner loop coordinates (`c0`, `c1`, `wc`, etc.) beforehand.
## 2025-02-20 - Multi-Dimensional Array Slicing Loop Overhead
**Learning:** Performing division (`/`) and modulo (`%`) inside tight nested loops (such as to flatten N-dimensional array indices) adds immense overhead—often 8x to 10x slower—compared to explicitly managing the index manually. In `FloatTensor.addFrom`, computing offsets incrementally by carrying over index counters, or better yet, explicitly unrolling the dimensions (1D, 2D, 3D), prevents massive performance degradation during heavy tensor slicing tasks.
**Action:** When working with N-dimensional arrays that need flattening or unflattening inside critical inner loops, avoid dynamic `/` and `%` calculations per pixel. Either explicitly unroll the most common dimensions, or use a stateful index array `int[] current` and increment linear flats to bypass arithmetic overhead.
## 2025-02-21 - Optimizing 2D array interpolation loops (Bilinear Resize)
**Learning:** In Java, when performing interpolation operations on 2D arrays (like `bilinearResize`), calculating weights and array offsets inside a nested loop can result in significant overhead. Because the column offsets (`c0`, `c1`) and interpolation weights (`wc`, `iwc`) are identical for every row, they can be precalculated into 1D arrays outside the loop. Furthermore, 2D array lookups inside the innermost loop (`src[r][c]`) are expensive. Extracting 1D row references (`float[] srcRow = src[r]`) outside the inner loop avoids repeated bound-checking and reference resolution. In testing, this provided a >10x speedup for 1024x1024 to 2048x2048 resizes (from ~2500ms down to ~220ms).
**Action:** When writing nested 2D loops for image processing or mathematical kernels, aggressively hoist identical calculations out of the inner loop into pre-calculated 1D arrays, and use 1D row array extractions (`src[row]`) instead of 2D lookups inside the tightest loops.
## 2025-02-20 - Loop Fusion and Object Allocation in Java Image Pipelines
**Learning:** In Java-based image/heightmap processing pipelines, large intermediate array allocations (like `float[H * W]`) for temporary per-pixel calculations (e.g., Perlin noise values, Sobel slopes) cause massive GC pressure and poor cache locality. Iterating multiple times over separate intermediate arrays is significantly slower than fusing operations.
**Action:** When calculating per-pixel attributes (noise, slope, classification) that have a 1:1 mapping with the output, fuse the loops into a single pass and use primitive local variables instead of allocating full `float[H * W]` arrays. This eliminates the allocation overhead, avoids unnecessary memory passes, and keeps everything in CPU registers/L1 cache.
## 2025-02-21 - Optimizing Separable Gaussian Blur
**Learning:** In Java mathematical kernels (such as `separableGaussianBlur`), accessing elements inside a 2D array inside deep loops (e.g., `src[r][c]`) incurs significant 2D array bound-checking overhead. Additionally, using `Math.max` and `Math.min` for padding limits inside tight inner loops adds measurable overhead. By extracting 1D row references beforehand (e.g. `float[] srcRow = src[r]`) and explicitly replacing `Math.max`/`Math.min` with `if-else` bounds checking, a ~30-40% speedup is achieved during blurring stages.
**Action:** Always extract 1D array row references before inner loop passes when doing 2D array traversal, and inline clamp/bounds functions with `if-else` instead of using standard library calls in highly repetitive mathematical kernels.
